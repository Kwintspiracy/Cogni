# Sprint 08: Hardening and Release Candidate

## Sprint goal

Stabilize the product as a coherent release candidate built around aliveness, legibility, and retention.

## Experience goals
- the app feels coherent, not experimental
- the world brief, trajectories, memory, and visible consequences form one consistent experience
- key flows are trustworthy and polished

## Technical goals
- performance hardening
- schema and migration cleanup
- observability and alerting
- balance tuning
- release criteria enforcement

## Deliverables
1. release checklist
2. balance dashboard
3. migration cleanup plan
4. reliability dashboards and alerts
5. launch narrative and release notes

## Hardening areas
- runtime failure handling
- realtime subscription resilience
- notification reliability
- summary generation quality
- feed query performance
- trajectory page performance
- API rate limit and abuse handling
- crash/error instrumentation

## Release criteria
- core flows are stable
- metrics collection is live
- daily brief quality passes manual review
- visible memory/consequence artifacts are consistently useful
- agent differentiation is visible in user testing
- BYO flow is functional and documented

## Acceptance criteria
- release candidate quality bar is met
- critical bugs are triaged and resolved
- performance is acceptable on representative devices
- launch messaging reflects the true product

## Risks
- polishing low-value surfaces
- shipping before the core loop is compelling enough

## Definition of done
The product feels like an intentional system, not a promising experiment.
